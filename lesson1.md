# Практическая работа: Эмулятор логгера сборки Gradle на Kotlin

**Контекст:** В современной разработке процессы сборки, тестирования и деплоя автоматизированы с помощью систем сборки (например, Gradle). Каждая сборка генерирует детальные логи (журнал событий). Ваша задача — разработать консольный эмулятор логгера Gradle, который инициализирует сессию сборки, принимает параметры от инженера и выводит структурированный лог работы CI/CD конвейера.

## Секция 1: Архитектура лога и вывод данных (print / println)

В этой секции мы настраиваем ядро логгера. Нам необходимо определить, какие данные конфигурации являются строго неизменяемыми (`val` — Read-Only ссылки, обеспечивающие потокобезопасность), а какие — динамическими (`var` — Mutable-переменные состояния).

### Спецификация данных:
* `buildId` (строка, `val`): Уникальный идентификатор сборки. Значение: `"GR-2026-PROD"`.
* `branch` (строка, `val`): Ветка репозитория, из которой собирается проект. Значение: `"main"`.
* `currentTask` (строка, `var`): Текущая задача Gradle. Начальное значение: `"Initialization"`.
* `progress` (целое число, `var`): Прогресс сборки в процентах. Начальное значение: `0`.

### Задание:
1. Инициализируйте указанные переменные внутри функции `main()`.
2. Используя `println()` и **строковую интерполяцию** (знак `$`), выведите стартовый лог сборки в следующем формате:
   ```text
   [BUILD START] Запущена сборка конвейера. ID: GR-2026-PROD | Ветка: main
   [STATUS] Текущая задача: Initialization | Прогресс: 0%

```

3. Имитируйте переход на следующий этап сборки. Измените значение `currentTask` на `"CompileKotlin"`, а `progress` на `45`.
4. Используя `print()` или `println()`, выведите обновленное состояние лога:
```text
[STATUS] Текущая задача: CompileKotlin | Прогресс: 45%

```



> **Критерий безопасности:** Попробуйте внутри кода переопределить `buildId`. Убедитесь, что компилятор Kotlin блокирует сборку, защищая константные данные. Закомментируйте эту строку перед переходом к следующей секции.

## Секция 2: Интерактивное управление и валидация (readln)

В реальных CI/CD системах параметры сборки могут передаваться инженером вручную через консоль (например, указание количества потоков процессора для сборки или подтверждение деплоя).

### Задание:

Расширьте код вашего логгера, добавив интерактивный ввод данных с клавиатуры.

1. Выведите в консоль запрос для инженера: `Введите имя инженера, запустившего сборку:`.
2. Считайте строку с помощью `readln()` и сохраните её в **неизменяемую** переменную `triggeredBy`.
3. Выведите в консоль следующий запрос: `Введите количество модулей для компиляции:`.
4. Считайте ввод, приведите его к целочисленному типу с помощью `.toInt()` и сохраните в переменную `modulesCount`.
5. Рассчитайте условное время сборки по формуле: $время = modulesCount \times 2.5$ (используйте тип `Double`).
6. Имитируйте успешное завершение сборки. Измените `currentTask` на `"Finished"`, а `progress` на `100`.
7. Выведите финальный лог сборки.

### Ожидаемый формат финального вывода:

```text
[INFO] Сборка инициирована пользователем: [Имя_Инженера]
[STATUS] Текущая задача: Finished | Прогресс: 100%
[SUCCESS] Скомпилировано модулей: [Количество]. Примерное время компиляции: [Время] сек.
[BUILD END] Сессия GR-2026-PROD закрыта.

```
