# Практическая работа №2: Интеллектуальный логгер Gradle (Ветвление `if` и `when`)

**Контекст:** На прошлом спринте мы написали линейный скрипт логгера, который слепо выводил статус `SUCCESS` независимо от происходящего. В реальном production-конвейере CI/CD система должна обладать «интеллектом»: анализировать код возврата тестов, замерять объем собранных файлов и переключать логику работы в зависимости от команд инженера. 

---

## Секция 1: Анализ стабильности сборки (Выражение `if / else`)

На этом этапе мы научим наш логгер реагировать на результаты автоматического тестирования кода и размер получившегося файла (артефакта).

### Спецификация данных:
Внутри функции `main()` объявите следующие переменные:
* `exitCode` (целое число, `val`): Код завершения тестов. Симулирует результат работы тестового движка. Значение для теста: `1` (где `0` — тесты пройдены, любое другое число — ошибка).
* `artifactSizeMb` (целое число, `val`): Размер собранного приложения. Значение для теста: `125`.

### Задание:
1. **Проверка тестов:** Используя конструкцию `if/else`, проверьте значение `exitCode`.
   * Если `exitCode == 0`, выведите в консоль: `[INFO] Все юнит-тесты успешно пройдены.`
   * Если `exitCode` равен любому другому числу, выведите: `[CRITICAL ERROR] Юнит-тесты провалены. Сборка аварийно остановлена!`
2. **Анализ размера артефакта (if как выражение):** Создайте переменную `val storageStatus: String`. Присвойте ей значение с помощью `if/else` как выражения:
   * Если `artifactSizeMb > 100`, выражение должно вернуть строку `"OVERSIZE"` (превышен лимит хранилища).
   * В противном случае — строку `"OK"` (размер в пределах нормы).
3. Выведите результат анализа размера в консоль в формате: `[STORAGE] Статус диска: [storageStatus] ([artifactSizeMb] MB)`.

---

## Секция 2: Интерактивный диспетчер стадий (Конструкция `when`)

На этой стадии инженер управляет конвейером через консоль. Нам нужно заменить громоздкие цепочки `if/else if` на профессиональный оператор множественного выбора `when`.

### Задание:
1. Выведите в консоль запрос для инженера: `Введите целевую команду (compile, test, deploy, clean):`.
2. Считайте ввод с клавиатуры с помощью `readln()`. Чтобы ввод не зависел от регистра букв (например, если студент введет `COMPILE` или `Compile`), примените к считанной строке встроенную функцию защиты: `readln().lowercase()`. Сохраните результат в переменную `command`.
3. Используя конструкцию `when(command)`, реализуйте следующий маршрутизатор логов:
   * `"clean"` $\rightarrow$ вывесть `[GRADLE] Очистка кэша и директории /build...`
   * `"compile"` $\rightarrow$ вывесть `[GRADLE] Компиляция исходного кода Kotlin...`
   * `"test"` $\rightarrow$ вывесть `[GRADLE] Запуск интеграционных тестов автоматизации...`
   * `"deploy"` $\rightarrow$ вывесть `[GRADLE] ВНИМАНИЕ: Отправка артефакта на продакшн-сервер!`
   * `else` (если введено что-то другое) $\rightarrow$ вывесть `[INVALID] Ошибка: Команда '[command]' не поддерживается автоматизацией.`

---

## Задание повышенной сложности (Task+)

Реализуйте сложную проверку безопасности для команды деплоя внутри блока `when`. 
Если инженер ввел команду `"deploy"`, программа должна дополнительно проверить переменную `exitCode` из первой секции:
* Если `exitCode == 0` (тесты чистые), разрешить деплой и вывести: `[DEPLOY] Статус: РАЗРЕШЕНО`.
* Если `exitCode != 0` (тесты упали), заблокировать операцию и вывести: `[DEPLOY] ОТКЛОНЕНО: Нельзя деплоить билд с упавшими тестами!`.
